## April 3, 2026 addendum: Problem 103 integration

The file now includes an additional scenario, `run_problem_103_delta_tracking()`, which maps the benchmark result to a finance use case.

1. `AtomicDeltaTracker` wraps `std::atomic<long long>`.
2. Fills are applied as signed quantity updates with `fetch_add(..., memory_order_relaxed)`.
3. A detector thread continuously reads the latest delta via relaxed loads.
4. No mutex is used between market-data writers and the detector reader.

This addendum demonstrates why the atomic benchmark result matters operationally: risk state can be observed continuously without lock contention.

## Concept Summary
- Without mutex: fastest-looking but wrong due to data race and lost increments.
- With mutex: correct, but each increment takes lock/unlock overhead.
- With atomic: correct and often faster than mutex for simple shared counters.

## Why memory_order_relaxed is okay here
- You only need atomicity of the increment itself, not ordering of unrelated data.
- The result is read after all threads have joined, so completion is already synchronized by thread lifecycle.
- Relaxed ordering minimizes overhead while preserving correct final count for this specific pattern.
