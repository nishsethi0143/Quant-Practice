## Big idea

`2.cpp` models two hot market-data paths updating in parallel:

1. call feed
2. put feed

If both feeds touch data on the same cache line, cores continuously invalidate each other and throughput drops. The file benchmarks that effect and compares cache-aware layouts.

## What 2.cpp compares

The benchmark runs two threads and each thread performs 10,000,000 atomic increments with `memory_order_relaxed`.

It compares three book layouts:

1. `SharedLineBook`
- `call_bid` and `put_bid` are tightly packed.
- High probability of destructive interference.
2. `CacheAlignedBook64`
- Uses `alignas(64)` for manual cache-line separation.
3. `CacheAlignedBookHw`
- Uses `alignas(std::hardware_destructive_interference_size)` for hardware-aware separation.

## Why relaxed atomics are used

The benchmark isolates cache-line contention effects, not ordering-heavy synchronization.

`memory_order_relaxed` keeps updates atomic while minimizing ordering overhead, making false sharing behavior easier to observe.

## Benchmark flow

`run_feed_benchmark()` does the same steps for each layout:

1. start timer
2. launch call feed thread
3. launch put feed thread
4. each thread updates only its own feed counter
5. join both threads
6. print elapsed milliseconds

At the end, `benchmark_false_sharing()` prints a `(C-P)` tick imbalance snapshot.

## Finance interpretation

1. Problem 102 (Put-Call Parity)
- Faster call/put updates improve how quickly parity imbalance can be measured.
2. Problem 104 (Digital Options)
- Reduced core contention lowers delay for near-synchronous feed processing.

## Practical takeaway

1. Layout can be as important as algorithm choice in low-latency systems.
2. Separating independently written hot fields often improves both throughput and stability.
3. Always benchmark on target hardware because scheduler and cache topology matter.
