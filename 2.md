# Deep Guide to 2.cpp: False Sharing and Cache-Line Aware Layout

Last updated: March 25, 2026

This document explains why `2.cpp` exists, how it works, and how to interpret its results.

## Big idea

`2.cpp` demonstrates false sharing.

False sharing happens when two threads write to different variables that live on the same cache line. Even though they do not touch the same variable, the CPU cache coherence protocol still invalidates and transfers that shared cache line between cores, causing heavy slowdown.

## What 2.cpp compares

The benchmark runs two threads, each incrementing its own atomic counter 10,000,000 times with `memory_order_relaxed`.

It compares three layouts:

1. `BadLayout`
   - `count1` and `count2` are close together.
   - They can land on the same cache line.
   - Usually the worst throughput.
2. `GoodLayout`
   - Both counters are `alignas(64)`.
   - On many modern CPUs, cache line size is 64 bytes, so this often separates them.
3. `OptimizedLayout`
   - Uses `alignas(std::hardware_destructive_interference_size)`.
   - This is a portable hint for cache-line separation according to platform/library constants.

## Why relaxed atomics are used

The benchmark is about cache-line movement, not ordering constraints between data structures.

Using `memory_order_relaxed` keeps operations atomic but avoids unnecessary ordering cost. This makes the false-sharing effect easier to observe.

## Walkthrough by section

## Includes

- `#include <atomic>`: atomic counters.
- `#include <chrono>`: timing.
- `#include <thread>` and `#include <vector>`: worker threads and containers.
- `#include <new>`: `std::hardware_destructive_interference_size`.

## Layout structs

- `BadLayout`: likely destructive interference.
- `GoodLayout`: manual padding/alignment strategy.
- `OptimizedLayout`: hardware-aware alignment strategy.

## Benchmark function

`benchmark_false_sharing()` does:

1. Construct layout object.
2. Start timer.
3. Launch two threads.
4. Each thread performs 10 million `fetch_add` operations on a different counter.
5. Join threads.
6. Stop timer and print milliseconds.

This sequence repeats for all three layouts.

## Interpreting results

Typical pattern:

- Bad layout is slower.
- Padded/hardware-aware layout is faster.

Do not overfit to one run. Variance comes from:

1. CPU frequency changes.
2. Thread scheduling noise.
3. Other processes on the machine.
4. Different compiler/optimization settings.

Run several times and compare trends.

## What this does NOT prove

1. It does not mean alignment always helps every workload.
2. It does not replace full profiling in production systems.
3. It does not cover NUMA effects or thread pinning.

## How to experiment safely

Try these controlled experiments in `2.cpp`:

1. Increase loop count to 50,000,000.
2. Test with 4 threads and 4 counters (extend layouts).
3. Compare `-O2` vs `-O3`.
4. Run multiple trials and average results.

## HFT relevance

Low-latency engines often process independent counters/queues per core. If hot fields for different threads share a cache line, throughput and tail latency can degrade significantly.

Avoiding false sharing is a practical, high-impact optimization in such systems.

## Quick summary

- False sharing is a hardware-level performance bug.
- Data layout can matter as much as algorithm choice.
- `2.cpp` gives a minimal, reproducible demonstration.
